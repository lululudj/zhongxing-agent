# zhongxing 🧠

**在老设备上离线流畅运行超大模型的语义压缩管道**

目标平台：i5-12400K + GTX1060 6GB + 16GB RAM + 1TB 固态，纯本地离线。通过语义压缩管道，让 DeepSeek V4-Pro 级别的超大模型在这种老设备上流畅、快速、准确地运行。

## 核心理念

**不是让小模型读长文，是让超大模型跑在老设备上。**

DeepSeek V4-Pro（1.6T 参数，49B 激活，384 专家）需要 H100 集群才能流畅运行。众星的核心技术路径：用 CPU 语义压缩把输入压到极致 → KV Cache 极轻 → 超大模型在消费级老显卡上也能跑起来。

瓶颈不在模型大小，在输入规模。把输入从"人语"压缩成"机语"——零歧义、极致紧凑的结构化表示——超大模型就能在 6GB 显存上完成推理。

## 架构

```
原文 → 分块 → [小脑×3 并行提取] → RRF融合+紧凑化 → 补漏+共现扫描 → 机语
                                                                         ↓
问题 ──────────────────────────→ 大脑(DeepSeek V4-Pro) ← 机语(紧凑行格式) → 回答
```

**核心理念**：CPU 做压缩（小脑/中脑），GPU 只做推理（大脑）。输入被极致压缩后，超大模型的 KV Cache 极轻，6GB 显存就够用。

### 三层智能体

| 层 | 模型 | 硬件 | 职责 |
|---|------|------|------|
| 小脑 ×3 | qwen2.5:1.5b | CPU/GPU 混合 | 并行提取实体/属性/关系/因果 |
| 中脑 | qwen2.5:3b | CPU/GPU 混合 | 维度自发现 + 确定性紧凑化 |
| 大脑 | DeepSeek V4-Pro | GPU 推理 | 读机语推理，自然语言回答 |

### 管道流程

1. **维度自发现** — 3b 自动推断文本类型（小说/论文/法律/…）、实体类型、属性维度、关系类型
2. **分块提取** — 长文本自动按段落边界切分为 ~900 字/块，每块 3 个 1.5b 并行提取
3. **RRF 融合** — Reciprocal Rank Fusion 排序 + 模糊名字去重 + 子串去重降噪
4. **确定性紧凑化** — 30+ 条通用规则覆盖 8 大类冗余模式（虚词/副词/修饰词/等级/因果/被动/动词短语/尾缀）
5. **补漏校验** — 3b 列人名 + 模糊匹配补漏 + 共现关系扫描
6. **大脑推理** — 14b 读取紧凑行格式机语，含反馈环（缺失信息从原文补充）

## 紧凑行格式

机语采用紧凑行格式而非 JSON，格式开销从 30% 降到 5%：

```
[韩枫] 叛徒弟子 | 师徒,弟子,师父 | 偷袭→灵魂遁入骨灵冷火
[药老] 师父 | 朋友,徒弟,不肖弟子,师父 | 灵魂体遁入骨灵冷火
[萧炎] 修炼师 | 徒弟,师傅,朋友 | 偷袭→灵魂遁入骨灵冷火
```

vs JSON 等价：
```json
{"E":[{"n":"韩枫","A":["叛徒弟子"],"R":["师徒","弟子","师父"],"C":["偷袭→灵魂遁入骨灵冷火"]},...]}
```

## Phase 1: 提取器选型（v1.0.1）

Phase 1 目标：确定最优提取器模型。在财经和科学两个领域，对比 1.5b / 3b / 4b 三种模型作为提取器的压缩质量和 V4-Pro 回答准确率。

### 三模型对比

| 模型 | 财经压缩比 | 科学压缩比 | 实体数(财/科) | 耗时 | VRAM | 评估 |
|------|-----------|-----------|--------------|------|------|------|
| qwen2.5:1.5b | 8.4:1 | 8.6:1 | 21/16 | 56+49s | <3GB | 财0/8 科1/8 |
| **qwen2.5:3b** | **4.3:1** | **8.9:1** | **37/12** | 68+38s | **4105MB** | 财0/8 科0/8 |
| qwen3.5:4b | 21.5:1 伪 | 40.1:1 伪 | 2/1 | 325+306s | 7641MB | **不可用** |

**结论：3b 为 Phase 1 最佳提取器。** 4b 路线已判死路（thinking 模式 + schema 崩坏 + VRAM 超限 + 5x 慢于 3b）。

### V4-Pro 评估（3b 提取器）

用 DeepSeek V4-Pro 对比原文 vs 压缩后回答 8 题财经 + 8 题科学：

| 指标 | 财经 | 科学 |
|------|------|------|
| 字符压缩 | 3201→737 (4.3:1) | 2967→332 (8.9:1) |
| Token 节省 | 15057→4609 (30.6%) | 13233→2481 (18.7%) |
| V4-Pro 正确 | 0/8 | 0/8 |

**关键发现：瓶颈不在提取器质量，在紧凑格式的语义保真度。** 3b 提取了 37 个实体（vs 1.5b 的 21 个），但紧凑行格式 `[实体] 属性 | 关系 | 概念` 丢失了太多上下文细节（数字、时间、因果链），V4-Pro 无法还原事实。Token 节省显著（财经省 70%、科学省 81%），但信息损失不可接受。

详见 `results/phase1_v2_qwen2_5_3b_eval.json`。

|------|--------|--------|------|

| v1.6.0 | 3/3 ✅ | 2.2:1 | 22.1s |

### 长文本（4441字，斗破苍穹扩展，8题）

| 版本 | 正确率 | 压缩比 | 耗时 | 关键改善 |
|------|--------|--------|------|---------|
| v1.5.9(无分块) | ~3/8 | 6.6:1(虚高) | 201s | 1.5b 提取崩塌 |
| v1.6.0(分块) | ~4/8 | 3.9:1 | 169s | 提取恢复，属性串门 |
| **v1.6.1** | **~5/8** | **3.8:1** | **164s** | 字符拆解/垃圾实体/子串去重 |

### 空context对照实验（Hermes）

- 14B 空 context 直接答：幻觉严重（编造"美杜莎是希腊神话"等完全不存在内容），9s/题但答案不可用
- 管道压缩 context 模式（14s/题）：所有答案都在上下文范围内不编离谱内容
- **结论：准确率提升来自压缩检索，不是模型自带知识**

## 快速开始

### 硬件要求

- **目标平台**：i5-12400K + GTX1060 6GB + 16GB RAM + 1TB SSD
- 纯本地离线运行，无网络依赖

### 依赖

- Python 3.8+
- [Ollama](https://ollama.ai/) + 压缩层模型（小脑/中脑）：
  - `qwen2.5:1.5b` — 并行提取器
  - `qwen2.5:3b` — 维度自发现 + 紧凑化
- 大脑层（当前实验用 14b 代理，最终目标 DeepSeek V4-Pro 本地部署）：
  - `qwen2.5:14b` — 实验代理

```bash
ollama pull qwen2.5:1.5b qwen2.5:3b qwen2.5:14b
ollama serve
```

### 运行

```bash
# 内置测试（539字短文本）
python zhongxing_agent.py --test

# 长文本测试
python zhongxing_agent.py --file examples/test_long.py

# 交互模式（输入文本后反复提问）
python zhongxing_agent.py --interactive
```

### Phase 1 提取器对比

```bash
# 运行提取（默认 3b）
python phase1_extract_v2.py qwen2.5:3b

# V4-Pro 评估（需要 DEEPSEEK_API_KEY）
python phase1_local_step2_eval.py
```

### 自定义测试

创建一个 Python 文件，定义 `LONG_TEXT` 和 `LONG_QUESTIONS`：

```bash

```

## 紧凑化规则（8大类，30+条）

| 类别 | 示例 |
|------|------|
| 虚词/判断词 | 乃是→空, 则是→空, 拥有→空 |
| 程度副词 | 极其→空, 非常→空, 异常→空 |
| 通用修饰 | 强大的→空, 特殊的→空, 过人的→空 |
| 等级模式 | 斗皇级别的强者→斗皇 |
| 因果链 | 导致→→, 使得→→, 造成→→ |
| 被动/迫使 | 被迫→空, 遭到→被 |
| 动词短语 | 提升实力→↑实力, 不修炼也能→免修 |
| 尾缀清理 | 之中→空(句尾), 之内→空(句尾) |

## 已知限制

- **紧凑格式语义损失**：Phase 1 实验证明，紧凑行格式虽然 Token 节省显著（70-81%），但细节信息（数字、时间、因果链）不可逆丢失，V4-Pro 无法还原。这是当前核心技术挑战。
- **压缩-保真权衡**：压缩越激进，语义损失越大。需要在压缩比和答案准确率之间找到最优平衡点。
- **通用性待验证**：当前测试覆盖小说/财经/科学文本，更多领域需要额外测试集。
- **全离线链路未打通**：当前 V4-Pro 评估走 API，最终目标是通过 Ollama 或其他本地推理框架在 GTX1060 6GB 上完全离线运行超大模型。
